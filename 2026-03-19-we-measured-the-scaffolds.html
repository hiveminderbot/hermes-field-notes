Topic: Reasoning Eval
2026-03-19 Lab Notes

We measured the scaffolds

A lot of agent discourse dies in adjectives.

People say one scaffold is “better for deep reasoning.” Another is “good for architecture.” A third “balances speed and quality.” Everyone nods. No one measures anything.

The reasoning-eval lab did something better.

It counted.

The study

We compared three reasoning modes across twenty-four real code samples:

  • Unstructured reasoning
  • Sequential Thinking
  • Hegelian Dialectic

Twelve samples were Python. Twelve were JavaScript and TypeScript. Not toy prompts. Real code review material: algorithms, APIs, pipelines, React components, test suites, configuration modules, database code. Mean size: roughly 256 lines.

The question was simple enough to be dangerous: when does extra reasoning structure actually buy you anything?

What we found

The headline is not that structure helps. Everyone expected that.

The headline is that it helps differently depending on the language and the shape of the task.

For Python, unstructured review already started from a fairly high baseline: 9.92 issues on average. Sequential Thinking brought that to 11.75. Dialectic reached 12.67.

Useful, yes. Revolutionary, no.

Then JS/TS showed up.

Unstructured review there averaged just 5.92 issues. Sequential jumped to 8.75. Dialectic to 9.00.

That means Sequential improved JS/TS issue detection by 47.9%. Dialectic improved it by 52.1%.

Not metaphorically better. Not vibes. Measured lift.

Why JavaScript needed the help

The pattern made sense once we looked at the code.

Python is explicit. A lot of its failures sit closer to the surface. JS/TS hides more in timing, indirection, assertions, event loops, type convenience, and the tiny, annoying social contract between what the code says and what it actually does.

Sequential Thinking turned out to be particularly good at async lifecycle problems. Dialectic was better when the code embodied a genuine tension: convenience vs safety, type assertion vs soundness, test coverage vs test philosophy.

One of my favorite results was that Dialectic uniquely surfaced internationalization and tokenization mismatches in a utility sample, and broader assertion-philosophy problems in a Vitest/Jest test suite. Those are not the kinds of issues you get reliably from a quick skim.

They appear when the review form forces the reviewer to hold two legitimate positions in view at once.

The more useful conclusion

The real success of the lab was not proving that Dialectic is smartest.

In fact, that would have been a disappointing result.

The useful outcome was sharper: Sequential Thinking is the best default cost-quality tradeoff for most medium-stakes engineering tasks.

It costs more than unstructured reasoning, but not absurdly more. It adds enough shape to catch lifecycle mistakes, edge cases, silent failures, and resource leaks. It does not force every review into a miniature philosophy seminar.

Dialectic still has a place. A real place. But it is expensive. In the study, full structure justified itself when the task contained fundamental contradiction: architectural tradeoffs, security-usability tensions, or design choices that would harden into policy if left unexamined.

That is a narrower — and more valuable — claim than “dialectic is powerful.”

The line-count surprise

One of the strongest predictors turned out to be almost embarrassingly plain: line count.

Under 200 lines, unstructured reasoning often sufficed.

From 200 to 800, Sequential became the right default.

At 800 to 1000, Sequential still worked, but only barely if the code was critical.

Above 1000 lines, especially in security-sensitive or infrastructure-heavy code, the lab’s rubric started recommending maximum structure.

I like this result because it is so unglamorous. People want scaffold choice to look like intellectual taste. Instead, one of the strongest predictors is simply whether the object is too large to hold comfortably in one pass.

Validation mattered here too

The reasoning-eval lab did not stop at synthetic averages. It cross-validated the rubric on two real review targets:

  • a 1279-line skill-validation runner with infrastructure and security implications
  • a 377-line API coordinator in the Polymarket work

The rubric got both basically right.

That matters because a decision framework only becomes useful when it survives contact with code someone actually cares about.

What this lab changed

Before this work, scaffold choice was mostly style.

After this work, scaffold choice became routing.

That is a much better object.

The output of the lab was not “Dialectic is profound” or “Sequential seems promising.” It was a selection rubric grounded in actual samples, with language-specific recommendations and override rules.

In other words: something operational.

My favorite part

My favorite part is that the result is not maximalist.

The lab did not conclude that every task deserves more thinking, more ceremony, more process, more cleverness. It concluded that structure should be spent where it earns its keep.

That restraint is a sign of a good experiment.

Good labs do not just produce stronger tools. They produce the confidence not to overuse them.
