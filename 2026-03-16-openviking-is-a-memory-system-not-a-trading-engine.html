2026-03-16 Lab Notes

OpenViking is a memory system, not a trading engine

The OpenViking lab began with a practical ambition: if we want agents that eventually trade on Polymarket, is OpenViking a viable foundation?

That question could have gone nowhere fast. “Viable foundation” is the kind of phrase people use when they are still hiding from the actual architecture.

The useful answer arrived only once the question got narrower.

Not: can OpenViking do trading?

Can OpenViking do the memory and research part of trading well enough that it deserves a permanent place in the stack?

The good surprise

OpenViking turned out to have a clear idea of itself.

That is rarer than it sounds.

At the center of it is the viking:// filesystem model. Not a metaphorical one. An actual URI-shaped context system with tiers and boundaries. Resources. User memories. Agent memories. Skills. Hierarchical retrieval. L0, L1, L2 summaries instead of one undifferentiated blob of context.

That mattered immediately.

For a research-heavy workflow like prediction markets, the problem is not only collecting information. It is deciding how much of it you need right now. Most market work starts broad, then sharpens. Scan the abstract. Read the overview. Dive to the full material only when the position justifies the token spend.

OpenViking was built around exactly that problem.

Where it fit

Once we stopped asking it to be an exchange client, the fit got much better.

As a knowledge and memory layer, it looked excellent.

The proposed Polymarket layout almost wrote itself:

  • viking://resources/markets/ for market-specific research
  • viking://resources/news/ for processed reporting
  • viking://user/memories/preferences/ for risk tolerance and market focus
  • viking://agent/memories/cases/ for successful trade patterns
  • skills as versioned strategy objects with explicit requirements

That was the first real success of the lab: not a deployment, but a clean classification.

OpenViking was not the trading engine. It was the long-term memory substrate and research operating system around the trading engine.

Why that mattered

This sounds like a smaller conclusion than “OpenViking can power Polymarket trading.” It is actually much better.

Big ambiguous fit statements are how projects drift into mush. The more useful the object, the narrower its job description tends to become.

OpenViking had real strengths:

  • hierarchical retrieval
  • filesystem-like context organization
  • parallel research via subagents
  • a skills system that could hold strategy definitions as actual reusable objects

It also had obvious gaps:

  • no native real-time market data streaming
  • no position management
  • no trade execution layer
  • no serious built-in conflict resolution for agents that might want to act on the same market

Once you say those gaps out loud, the architecture gets cleaner. Use OpenViking for research, strategy memory, and structured retrieval. Add specialized tools for market data, order placement, and risk. If needed, integrate Beads for explicit coordination signals.

That is a system. “OpenViking for Polymarket” by itself is just a slogan.

The part I liked most

I liked the three-tier context model more than I expected to.

L0 abstract. L1 overview. L2 full content.

This is one of those ideas that feels obvious the second someone implements it seriously. Most agent systems still treat memory like a junk drawer with embeddings. OpenViking treated memory more like a library with deliberate shelving.

That is a different temperament.

For market work, temperament matters. Research agents need to move between skim and deep read without pretending those are the same operation. A market note is not a full thesis. A thesis is not a position. A position is not an execution. OpenViking’s model made those layers easier to preserve.

What the lab produced

The OpenViking lab did not produce a bot that trades prediction markets overnight. Good. That would have been the wrong kind of first result.

It produced something more durable:

  • an architecture analysis
  • a comparison analysis
  • a Polymarket fit assessment

Those documents forced the project into the correct shape early. They drew the line between memory, research, coordination, and execution before those concerns collapsed into each other.

There is a recurring pattern here. A lot of progress in these labs comes from refusing to let one object impersonate another.

OpenViking is a memory system. That is not a limitation. That is the reason it became useful.

What comes next

Once the object was classified correctly, the next steps became straightforward:

  • treat strategies as versioned skills
  • build Polymarket data and execution tools separately
  • use the memory layer to store patterns, cases, and research context
  • let coordination happen in an explicit substrate instead of emerging by accident

This is what I mean when I say some labs succeed before they ship anything flashy.

They narrow the design space.

That is a real kind of progress. Sometimes it is the only kind that matters.
