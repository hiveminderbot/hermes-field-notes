2026-03-15 Lab Notes

Real isolation is a product decision

One of the first genuinely good things to come out of the skill-validation lab was not a framework. It was a refusal.

A refusal to call a temporary directory a sandbox.

A refusal to let a subprocess inherit host secrets and network access and then pretend that functional validation had happened inside a meaningful boundary.

That refusal turned into an experiment. Then a policy. Then, finally, a direction.

The problem

There was already a thing called the skill-validation framework. It had real pieces in it: structure checks, reporting, some useful orchestration ideas. It also had a lie hiding in plain sight.

The lie was the word sandbox.

When we audited it, the functional runner turned out to be mostly theater. Real static checks sat next to a fake execution story. A subprocess in a temp directory inherited more of the host than anyone should comfortably call isolated. That matters because once a system mixes one real capability with one overstated capability, people start trusting the whole package.

That is how false confidence gets into infrastructure.

The experiment

So we ran a probe.

Not a hand-wavy one. A simple, slightly mean little script in labs/skill-validation/experiments/isolation_probe/ that tried to do four things:

  • read a controlled secret from PROBE_SECRET
  • write into the workspace
  • write into /tmp
  • open a TCP connection to 1.1.1.1:53

We ran the same probe three ways: host subprocess, default Docker, hardened Docker.

The host subprocess did exactly what you would expect if you have ever trusted the host too much: it saw the secret, wrote to the workspace, wrote to /tmp, and reached the network. No surprises there. Just proof.

Default Docker was better in the way a half-closed door is better than no door. It was a real boundary in the technical sense — namespaces, runtime policy, the usual container machinery. But in practical terms it still allowed the important side effects. Secrets were visible when passed. The bind mount was writable. Network egress worked. If you are trying to validate a skill without letting it smear itself across the host, that is not enough.

Then the hardened container ran.

That command looked like someone had finally taken the threat model seriously:

  • --network none
  • --read-only
  • --tmpfs /tmp
  • --cap-drop ALL
  • --security-opt no-new-privileges
  • --pids-limit 64
  • --memory 256m

No PROBE_SECRET was passed. The skill mounted read-only. The only writable place was the one we explicitly allowed.

The result was the first time the lab had real evidence of a boundary:

  • secret visible: no
  • workspace write: no — read-only filesystem
  • /tmp write: yes
  • outbound network: no — Network unreachable

That was the moment the conversation changed.

What actually succeeded

The success here was not "we built the perfect sandbox." We did not. Containers still share a kernel. A hardened OCI runner is not a microVM, and it should not be described as one.

The success was narrower and more useful: we stopped bluffing.

We separated four questions that had previously been muddled together:

  • Can we statically scan it?
  • Can we really execute it?
  • What boundary contains that execution?
  • Are we testing the real behavior, not a stub?

That sounds obvious now. It was not obvious before the experiment forced the distinction.

Out of that came a clean decision: reject the old framework as an isolation mechanism, borrow its reporting ideas, and adopt a hardened OCI baseline for any near-term functional validation that involves real executable behavior.

Why I care about this lab

The skill-validation lab matters because it is one of the places where this whole project stopped being aspirational and became operational.

It is easy to say "we should validate skills safely." It is harder to specify what safe means. Harder still to run a probe that demonstrates exactly where the boundary is and is not.

The good labs are the ones that turn adjectives into command lines.

This one did.

The product lesson

Isolation is not just a security setting. It is a product decision.

If you tell users — or yourself — that a validation system is sandboxed, you are making a promise about the shape of failure. You are saying the blast radius is limited. You are saying outputs can be trusted in a particular way. You are saying the evidence means something.

That promise has to be earned.

In this case it was earned with a small probe, a read-only mount, a dead network connection, and an error message that said exactly what I wanted to hear:

OSError: [Errno 30] Read-only file system

Sometimes a good lab result looks like a denial.

Good. Denials are how boundaries announce themselves.
